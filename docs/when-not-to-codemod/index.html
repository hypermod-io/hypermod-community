When not to codemod

Codemods can't solve every problem, sometimes there is simply too much human intervention required.

Here is a list of some user cases that are not possible or is extremely hard to codemod:

When a codemod doesn't make sense?#

  • API / feature removal: cannot shift people to things that do not exist
  • When there is too much human intervention required. (consider writing a migration guide instead)
  • Changes that need an awareness of runtime usage (e.g. what is in the full result react tree)
  • Sometimes we need to know what values are being used. This can be hard if it is coming in across module boundaries, using object spreading and so on.
  • Usage paradigm shifts where the old paradigm does not have a 1:1 in the new paradigm
  • Cases where consumers need to provide more information than they did before (eg when a provided component now needs to provide a ref)

What to do when a codemod isn't possible?#

In most cases, you should aim to do as much of the work, and prompt for human input where possible.
